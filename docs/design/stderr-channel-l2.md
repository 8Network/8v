Column meanings:\n- **Channel:** where the text lands today (`stdout` or `stderr`)\n- **Correct?** whether it matches the target contract (human errors to stderr, JSON errors to stdout, normal output to stdout)\n- **Mechanism:** how text reaches the channel\n\n### 1.1 main.rs — the router\n\n| Lines | Path | Channel | Correct? Target Architecture\n\n### 2.1 The fix — one line\n\nFile: `o8v/src/commands/mod.rs`, line 252.\n\nBefore:\n```rust\nlet output = upgrade.run(&upgrade_args).await?;\nOk((render(output, audience)?, ExitCode::SUCCESS, false))\n```\n\nAfter:\n```rust\nlet output = upgrade.run(&upgrade_args).await?;\nOk((render(output, audience)?, ExitCode::SUCCESS, audience == Audience::Human))\n```\n\nThis is the same pattern already used by `check` (line 206), `fmt` (line 221), and `hooks` (line 245).\n\n### 2.2 What changes at runtime\n\n| Scenario | Before fix | After fix |\n|----------|-----------|----------|\n| `8v upgrade` (human, upgrade fails) | Error text on stdout | Error text on stderr |\n| `8v upgrade --json` (upgrade fails) | `{"error": "..."}` on stdout | `{"error": "..."}` on stdout |\n| `8v upgrade` (human, success) | Success text on stdout | Success text on stderr |\n| `8v upgrade --json` (success) | `{"upgraded": true, ...}` on stdout | `{"upgraded": true, ...}` on stdout |\n| `8v upgrade` via MCP | `use_stderr=true` triggers `Err(out)` in handler | Same — already handled by handler.rs:171-184 |\n\nNote on success output routing: After the fix, human-mode upgrade success text will also go to stderr. This matches the behavior of `check`, `fmt`, and `hooks` — all route their entire rendered output to stderr when audience is Human. These are diagnostic commands; their full output belongs on stderr so callers can separate signal from noise.\n\n### 2.3 No new types, no new traits, no bypass architecture\n\nThe fix does not:\n- Add a new `Output` variant or wrapper type\n- Change any render function signature\n- Add any conditional formatting logic\n- Create a new struct field\n\nThe render boundary is preserved: `upgrade_report.rs` still returns a plain `Output` struct; routing happens in `commands/mod.rs` as for every other command.\n\n### 2.4 Deeper cleanup — explicitly OUT OF SCOPE for B2a\n\nThe `UpgradeReport.error: Option<String>` field embeds an error in a success struct. This is an architectural smell (B2c scope). B2a does not touch it because:\n1. Removing the field requires a separate data-model change and render rewrite\n2. The 1-line `use_stderr` fix is sufficient to move the error to stderr\n3. JSON output with `"error": "..."` remains correct either way\n\nB2c will convert the `Ok(UpgradeReport { error: Some(e) })` path to `Err(CommandError::Execution(e))`.\n\n---\n\n## 3. Failing-First Tests\n\nEach test must be written before the code change and must FAIL against the unmodified binary. After the 1-line fix, all must pass.\n\n### 3.1 Test location\n\n`o8v/tests/upgrade_stderr.rs` (new integration test file)\n\n### 3.2 Test list\n\n```\nupgrade_human_error_goes_to_stderr\nupgrade_human_error_not_on_stdout\nupgrade_json_error_stays_on_stdout\nupgrade_json_error_not_on_stderr\nupgrade_human_success_goes_to_stderr\n```\n\n### 3.3 Test contracts\n\n**upgrade_human_error_goes_to_stderr**\n- Invoke: `8v upgrade --check-only` with an unreachable server URL\n- Assert: stderr contains `"error"` (case-insensitive)\n- Fails before fix: output is on stdout, stderr is empty\n\n**upgrade_human_error_not_on_stdout**\n- Invoke: same conditions as above\n- Assert: stdout does NOT contain `"error"`\n- Fails before fix: stdout contains the error text\n\n**upgrade_json_error_stays_on_stdout**\n- Invoke: `8v upgrade --check-only --json` with unreachable server\n- Assert: stdout parses as valid JSON containing `"error"` key with non-null value\n- Passes before fix (JSON was always correct) — included to guard against regression\n\n**upgrade_json_error_not_on_stderr**\n- Invoke: `8v upgrade --check-only --json` with unreachable server\n- Assert: stderr is empty or does not contain JSON\n- Passes before fix — regression guard\n\n**upgrade_human_success_goes_to_stderr**\n- Invoke: `8v upgrade --check-only` with a fixture server returning the current version\n- Assert: stderr contains version string\n- Assert: stdout is empty\n- Fails before fix: output is on stdout\n\n### 3.4 Test infrastructure notes\n\n- Use `std::process::Command` to spawn the real binary (out-of-process)\n- Capture stdout and stderr separately via `.stdout(Stdio::piped()).stderr(Stdio::piped())`\n- For the unreachable server: use a localhost port that is guaranteed closed (`127.0.0.1:1`) or pass an invalid base URL via the env var the upgrade command accepts for test injection\n- Do not mock binary internals — the test must invoke the real binary to validate channel routing through `main.rs`\n\n---\n\n## 4. Mutation Audit Plan\n\nFor each test, one single-line mutation must break exactly that test and no others. This proves the test catches the behavior it claims to catch.\n\n| Test | Mutation | Expected breakage |\n|------|----------|------------------|\n| `upgrade_human_error_goes_to_stderr` | Revert fix: change `audience == Audience::Human` back to `false` | Test fails: stderr empty |\n| `upgrade_human_error_not_on_stdout` | Same revert | Test fails: stdout contains error |\n| `upgrade_json_error_stays_on_stdout` | In JSON render, set `error: None` before render call | Test fails: JSON missing `"error"` key |\n| `upgrade_json_error_not_on_stderr` | Add `eprintln!("{:?}", report)` before render in upgrade arm | Test fails: stderr non-empty |\n| `upgrade_human_success_goes_to_stderr` | Revert fix | Test fails: stdout non-empty, stderr empty |\n\nMutation discipline: apply each mutation, run only the corresponding test, confirm it fails, then restore. Do not run the full suite during mutation audit — mutation is per-test verification, not a regression run.\n\n---\n\n## 5. Counterexample Cross-Check\n\nCounterexamples from `docs/design/stderr-channel-counterexamples.md`, with resolution status for the L2 design.\n\n### A2: JSON output must stay on stdout\n\nStatus: RESOLVED.\nThe fix sets `use_stderr = audience == Audience::Human`. No change to pass-through subprocess I/O.\n\n### A6: Event-log emitter path (trace required before shipping)\n\nStatus: CLOSED 2026-04-20 — clean (traced event_bus, storage_subscriber, tracing subscriber; all error paths use tracing::* → stderr or return Result; no println!/print! in emit paths)\n\n### A7: MCP handler channel\n\nStatus: RESOLVED — CLEAN.\nSurvey of `mcp/handler.rs:171-184` confirmed: all paths return `Err(String)` which the MCP framework converts to `is_error: true`. No writes to stdout. The `use_stderr` bool is inspected at line 171 and, when true, routes to `Err(out)` — also correct. B2a does not change the MCP handler.\n\n### A8: Render function topology\n\nStatus: RESOLVED — ONE BYPASS FOUND.\nFull survey of all render functions confirmed: only `upgrade_report.rs` embeds error text in the success output. All other render functions produce clean output. The bypass is exactly as described in the L1 doc. B2a does not need to touch any other render function.\n\n### A12: BrokenPipe on stderr write\n\nStatus: NOTED — NOT IN SCOPE FOR B2a.\nIf stderr is closed (e.g., `8v upgrade 2>/dev/null`), `eprintln!` will encounter BrokenPipe. This is a pre-existing condition shared by all `eprintln!` calls in the binary. B2a does not make this worse. BrokenPipe hardening is a separate infrastructure slice.\n\n### A14: In-process test capture\n\nStatus: RESOLVED — DESIGN DECISION.\nTests use `std::process::Command` to spawn the real binary (out-of-process). This sidesteps in-process stdout/stderr capture complexity entirely. The test infrastructure in section 3.4 reflects this decision.\n\n---\n\n## 6.
